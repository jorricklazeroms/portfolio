11:T6da,<h2>Problem</h2>
<p>We wilden een portfolio dat niet alleen projecten toont, maar ook teamverantwoordelijkheid en delivery-discipline zichtbaar maakt voor elke case.</p>
<h2>Constraints</h2>
<ul>
<li>Vaste stack: Next.js App Router + TypeScript.</li>
<li>Content moest in platte bestanden staan voor snelle iteratie.</li>
<li>Rollen moesten consistent zijn over projectcards, detailpagina's en devlogs.</li>
</ul>
<h2>Solution</h2>
<p>We hebben een contentmodel opgezet met <code>content/projects.json</code> (roles + accountability) en <code>content/devlog/*.md</code> (context per release). Projectdetailpagina's linken direct naar de juiste devlog-entry.</p>
<h2>Architecture</h2>
<ul>
<li>Routing: <code>app/projects/[slug]</code> en <code>app/devlog/[slug]</code>.</li>
<li>Validatie: Zod-schema's in <code>src/lib/projects.ts</code> en <code>src/lib/devlog.ts</code>.</li>
<li>Rendering: frontmatter + Markdown naar HTML via remark/rehype pipeline.</li>
</ul>
<h2>Collaboration Model</h2>
<ul>
<li>Technical Owner: Jorrick</li>
<li>Supporting Engineer: Wesley</li>
<li>Reviewer: Wesley</li>
<li>QA Owner: Jorrick</li>
</ul>
<h2>Testing and Quality Gates</h2>
<ul>
<li>Frontend gates: <code>npm run lint</code>, <code>npm test</code>, <code>npm run build</code>.</li>
<li>CI-doel: blokkeren op type/content-validatie en mislukte test/lint-stappen.</li>
<li>Devlog-links worden gecontroleerd via statische routegeneratie op slug-niveau.</li>
</ul>
<h2>Next Steps</h2>

</ul>
